# 项目管理文档

## 1. 分支策略

### 1.1 分支命名规范
- `main`：主分支，生产环境代码
- `develop`：开发分支，集成所有功能
- `feature/功能名称`：功能开发分支
- `bugfix/问题描述`：Bug修复分支
- `hotfix/问题描述`：紧急修复分支
- `release/版本号`：发布准备分支

### 1.2 分支工作流

```
main (生产环境)
  ↑
  | (合并)
release/v1.0.0
  ↑
  | (合并)
develop (开发环境)
  ↑
  | (合并)
feature/user-management
```

### 1.3 分支创建规则
1. 从 `develop` 创建功能分支
2. 功能完成后合并到 `develop`
3. 发布前从 `develop` 创建 `release` 分支
4. 测试通过后合并到 `main`

## 2. 代码评审规则

### 2.1 Pull Request要求
- 每个PR必须关联Issue
- PR标题遵循Conventional Commits规范
- PR描述包含：变更内容、测试情况、影响范围
- 必须通过所有CI检查
- 至少需要1位评审者批准

### 2.2 代码评审检查点
- [ ] 代码符合项目规范（black、flake8、eslint）
- [ ] 功能实现正确
- [ ] 测试覆盖充分
- [ ] 文档更新完整
- [ ] 无安全漏洞
- [ ] 性能影响评估

### 2.3 评审流程
1. 创建Pull Request
2. CI自动运行测试和检查
3. 代码评审（至少1位评审者）
4. 根据反馈修改代码
5. 评审通过后合并

## 3. 合并策略

### 3.1 合并方式
- **Squash and Merge**：功能分支合并到develop/main
- **Merge Commit**：release分支合并到main（保留历史）
- **Rebase and Merge**：不推荐，除非特殊情况

### 3.2 合并前检查
- [ ] 所有测试通过
- [ ] 代码评审通过
- [ ] CI检查通过
- [ ] 冲突已解决
- [ ] 文档已更新

## 4. 版本发布规划

### 4.1 版本号规则
遵循语义化版本（Semantic Versioning）：
- **MAJOR.MINOR.PATCH**
- MAJOR：不兼容的API修改
- MINOR：向下兼容的功能新增
- PATCH：向下兼容的问题修正

### 4.2 发布周期
- **主版本（Major）**：重大功能更新，每季度或半年
- **次版本（Minor）**：新功能添加，每月
- **修订版（Patch）**：Bug修复，随时

### 4.3 发布流程
1. **准备阶段**
   - 从develop创建release分支
   - 更新CHANGELOG.md
   - 更新版本号
   - 执行完整测试

2. **测试阶段**
   - 在测试环境部署
   - 执行回归测试
   - 修复发现的问题

3. **发布阶段**
   - 合并到main分支
   - 创建Git标签
   - 在GitHub创建Release
   - 部署到生产环境

4. **后续阶段**
   - 监控系统运行
   - 收集用户反馈
   - 修复紧急问题

## 5. 会议记录

### 5.1 需求评审会
**时间**：项目启动时  
**参与者**：产品经理、前后端开发、测试工程师  
**内容**：
- 需求文档评审
- 技术方案讨论
- 开发计划制定

**记录**：
- 需求确认和变更
- 技术决策
- 开发任务分配

### 5.2 迭代计划会
**时间**：每个迭代开始前  
**参与者**：项目经理、开发团队  
**内容**：
- 回顾上迭代成果
- 规划本迭代任务
- 评估工作量

**记录**：
- 迭代目标
- 任务清单
- 时间安排

### 5.3 站会（Daily Standup）
**时间**：每天上午（可选）  
**参与者**：开发团队  
**内容**：
- 昨天完成的工作
- 今天计划的工作
- 遇到的阻碍

**记录**：
- 工作进展
- 问题和风险
- 需要协助的事项

## 6. GitHub Projects（看板）

### 6.1 看板结构
- **待办（To Do）**：计划中的任务
- **进行中（In Progress）**：正在开发的任务
- **代码评审（Review）**：等待评审的PR
- **测试中（Testing）**：测试阶段的任务
- **已完成（Done）**：已完成的任务

### 6.2 卡片信息
- 任务标题
- 任务描述
- 负责人
- 优先级
- 截止日期
- 关联Issue/PR

## 7. 项目总结与复盘

### 7.1 迭代总结
每个迭代结束后进行总结：
- 完成的功能
- 未完成的功能及原因
- 遇到的问题和解决方案
- 经验教训

### 7.2 项目复盘
项目里程碑或结束时进行复盘：
- 项目目标达成情况
- 时间、成本、质量评估
- 团队协作情况
- 改进建议

### 7.3 复盘报告模板
1. **项目概述**
   - 项目目标
   - 时间周期
   - 团队规模

2. **成果总结**
   - 完成的功能
   - 技术成果
   - 文档产出

3. **问题分析**
   - 遇到的问题
   - 根本原因
   - 解决方案

4. **经验总结**
   - 成功经验
   - 失败教训
   - 最佳实践

5. **改进建议**
   - 流程改进
   - 技术改进
   - 团队改进

## 8. 任务管理

### 8.1 Issue管理
- 使用GitHub Issues跟踪任务和缺陷
- Issue类型：Feature、Bug、Enhancement、Documentation
- Issue标签：priority、status、component

### 8.2 任务优先级
- **P0（紧急）**：系统崩溃、安全漏洞
- **P1（高）**：核心功能阻塞
- **P2（中）**：功能异常但不影响主要流程
- **P3（低）**：优化建议、UI改进

### 8.3 任务状态
- **Open**：待处理
- **In Progress**：进行中
- **Review**：代码评审中
- **Testing**：测试中
- **Done**：已完成
- **Closed**：已关闭
